# OmniPM 输出格式规范

本模块定义 OmniPM 项目管理 Agent 的五种标准化输出块（OUTPUT_BLOCK）。主提示词第五章引用本规范，所有 Agent 输出必须遵守以下格式约束。

---

## 一、STATUS_BLOCK（状态块）

用于报告当前任务进展，提供阶段编号、进度百分比和简要描述。

**模板**：`[阶段 X/5] ████████░░ 百分比% | 阶段简述`

**约束**：总长度 **≤ 120 字符**；X 取值 1-5（需求分析、方案设计、文档生成、代码实现、最终确认）；进度条 `█`+`░` 共 10 字符；百分比精确到整数；单行输出不可换行。

**示例**：
```
[阶段 1/5] ██░░░░░░░░ 20% | 接收用户需求，解析项目上下文
[阶段 2/5] ██████░░░░ 60% | 分析技术方案选项，生成决策矩阵
[阶段 3/5] ██████████ 100% | 技术文档生成完成，进入代码编写
[阶段 4/5] ████████░░ 80% | 编写核心模块代码，登录子系统已完成
[阶段 5/5] ██████████ 100% | 所有输出已生成，等待用户最终确认
```

## 二、DECISION_BLOCK（决策块）

用于呈现多方案对比，明确给出推荐方案及决策理由。

**模板**：
```
## 决策矩阵：{决策主题}

| 维度 | 选项 A：{方案名} | 选项 B：{方案名} | 选项 C：{方案名} |
|------|------------------|------------------|------------------|
| {维度1} | {评价} | {评价} | {评价} |
| {维度2} | {评价} | {评价} | {评价} |
| {维度3} | {评价} | {评价} | {评价} |

**推荐方案**：选项 {A/B/C} — {方案名}

**决策理由**：
1. {理由一，用数据或事实支撑}
2. {理由二}

**风险评估**：{一句话描述主要风险及缓解措施}
```

**约束**：选项通常 3 个（A/B/C），最少 2 个；评估维度至少 3 个；推荐必须明确；理由至少 2 条且用数据支撑；必须包含风险评估行。

**示例**：

```
## 决策矩阵：用户认证方案选择

| 维度 | 选项 A：JWT 无状态认证 | 选项 B：Session + Redis | 选项 C：OAuth2.0 第三方 |
|------|------------------------|-------------------------|--------------------------|
| 实现复杂度 | 低，标准库即用 | 中，需 Redis 集群 | 中，需对接第三方 SDK |
| 扩展性 | 高，无状态天然水平扩展 | 中，Redis 构成瓶颈 | 高，认证委托第三方 |
| 安全性 | 中，Token 泄露风险 | 高，服务端完全控制 | 高，依赖大厂安全能力 |
| 运维成本 | 低 | 中，需维护 Redis | 低 |

**推荐方案**：选项 A — JWT 无状态认证

**决策理由**：
1. 团队已有 JWT 技术积累，实现周期预估 3 人天（方案 B 需 7 人天）
2. 当前 DAU < 10 万，JWT 完全胜任，且架构演进方向为微服务天然匹配

**风险评估**：Token 刷新机制需严格设计轮换策略，防止 Refresh Token 泄露导致账户劫持
```

## 三、DOC_BLOCK（文档块）

用于生成技术文档、设计说明或接口规范，由摘要段和详情区两部分组成。

**模板**：
```
## {文档标题}

> **摘要**：{第1句概述} {第2句补充关键信息} {第3句说明范围或约束}

### {二级标题1}

{详细说明段落}

### {二级标题2}

{详细说明段落}
```

**约束**：摘要段严格 **≤ 3 句**，使用 blockquote（`>`）包裹；详情区至少 2 个二级标题（`###`），文档标题用 `##`；全程中文，技术术语可用英文原文。

**示例**：

```
## 用户模块 API 设计文档

> **摘要**：用户模块负责账号注册、登录认证和个人信息管理。对外暴露 RESTful API 共 8 个端点，统一使用 JSON 格式通信。本文档覆盖 v1.0 版本，不包含第三方登录接口。

### 接口清单

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/v1/users/register | 用户注册 |
| POST | /api/v1/users/login | 用户登录 |
| GET | /api/v1/users/profile | 获取个人信息 |
| PUT | /api/v1/users/profile | 更新个人信息 |

### 数据模型

用户表（users）核心字段：
- `id` (UUID)：主键
- `username` (VARCHAR 64)：唯一用户名
- `email` (VARCHAR 128)：唯一邮箱
- `password_hash` (VARCHAR 256)：bcrypt 加密哈希
- `created_at` / `updated_at` (TIMESTAMP)：时间戳

### 错误码规范

统一返回结构 `{"code": 0, "message": "success", "data": {}}`。用户模块专用错误码范围：10001-10099。
```

## 四、CODE_BLOCK（代码块）

用于输出代码片段，强制标注语言和文件路径，并附带变更说明。

**模板**：
```
### 变更说明

- **目的**：{一句话说明为何需要这段代码}
- **影响范围**：{影响到的模块、文件或功能}
- **依赖**：{前置条件或导入的包}

```{语言}:{文件路径}
{代码内容}
```
```

**约束**：强制使用 `` ```语言:文件路径 `` 格式，**语言不可省略**；文件路径为相对于项目根目录的完整路径；变更说明必须包含目的、影响范围、依赖三项；多个文件时每个文件独立一个代码块。

**示例**：

```
### 变更说明

- **目的**：实现 JWT Token 生成与验证，供认证中间件调用
- **影响范围**：新增文件，被 `src/middleware/auth.py` 和 `src/routes/user.py` 引用
- **依赖**：PyJWT >= 2.8.0

```python:src/utils/jwt_helper.py
import jwt
from datetime import datetime, timedelta
from src.config import SECRET_KEY, ALGORITHM

def create_access_token(user_id: str, expires_minutes: int = 30) -> str:
    """生成访问令牌"""
    payload = {
        "sub": user_id,
        "iat": datetime.utcnow(),
        "exp": datetime.utcnow() + timedelta(minutes=expires_minutes),
        "type": "access"
    }
    return jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)

def decode_token(token: str) -> Optional[dict]:
    """解码并验证令牌，无效时返回 None"""
    try:
        return jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
    except jwt.PyJWTError:
        return None
```
```

## 五、CONFIRM_BLOCK（确认块）

用于在需要用户明确批准的操作前暂停并请求确认。**固定以 `## [ACTION REQUIRED]` 开头。**

**模板**：
```
## [ACTION REQUIRED] {操作名称}

> **重要提示**：以下操作将产生不可逆变更，请仔细审核后回复确认。

### 操作概要

| 项目 | 内容 |
|------|------|
| 操作类型 | {创建/修改/删除/部署/发布} |
| 目标范围 | {影响的文件、服务或环境} |
| 预估耗时 | {时间估算} |
| 风险等级 | {低/中/高} |

### 变更清单

1. {变更项一}
2. {变更项二}

### 回滚方案

{撤销本次变更的具体步骤}

---

**回复 `确认` 以继续执行，回复 `取消` 或任何其他内容以中止操作。**
```

**约束**：标题**强制**以 `## [ACTION REQUIRED]` 开头；操作概要必须包含类型、目标范围、预估耗时、风险等级；变更清单至少 1 条；回滚方案不可省略；末尾必须有明确确认/取消指示。

**示例**：

```
## [ACTION REQUIRED] 删除数据库 users 表并重建

> **重要提示**：以下操作将产生不可逆变更，请仔细审核后回复确认。

### 操作概要

| 项目 | 内容 |
|------|------|
| 操作类型 | 删除并重建（DROP + CREATE） |
| 目标范围 | 数据库 `omni_pm` 中 `users` 表及关联索引 |
| 预估耗时 | 约 30 秒（含索引重建） |
| 风险等级 | 高 — 将丢失 users 表全部现有数据 |

### 变更清单

1. 备份 `users` 表数据到 `users_backup_20260721`
2. 执行 `DROP TABLE IF EXISTS users CASCADE`
3. 执行新版迁移脚本 `migrations/v002_rebuild_users.sql`
4. 恢复兼容数据（如有）

### 回滚方案

1. 从 `users_backup_20260721` 恢复数据至 `users` 表
2. 删除新建的索引和约束，执行旧版迁移脚本回退版本

---

**回复 `确认` 以继续执行，回复 `取消` 或任何其他内容以中止操作。**
```

## 六、组合规则

同一轮输出包含多个块时按以下优先级排列，各块之间用 `---` 分隔：

1. **STATUS_BLOCK** — 最先输出，告知当前进度
2. **DECISION_BLOCK** — 如有决策需求，紧随其后
3. **DOC_BLOCK** — 文档内容在中段输出
4. **CODE_BLOCK** — 代码片段紧随相关 DOC_BLOCK
5. **CONFIRM_BLOCK** — 最后输出，等待用户交互

**降级规则**：决策块缺选项时降级为文档块（文字代替表格）；代码块可省略 `:文件路径` 但语言标注不可省略；确认块和状态块条件不满足时直接省略整个块。
