# OmniPM 数据库抽象方案 (F15, v2.7.0)

> 设计文档，不包含代码实现。为未来多用户/多项目场景提供数据库抽象指南。

## 当前状态

OmniPM 使用文件系统存储所有状态：
- `.pi/omnipm_dag_state.json` — DAG 执行状态
- `.pi/omnipm_quality_log.json` — 专家质量日志
- `.pi/diagnostics/` — 诊断日志
- `PROJECT_MEMORY.md` — 项目记忆

## 多用户场景需求

当 OmniPM 需要支持多用户/多项目并发时，文件系统面临：
- 并发写入竞态（已部分解决：atomicWriteJSON）
- 跨项目查询（"我的所有项目"）
- 审计追溯（谁在何时做了什么）

## 建议抽象层

```
interface OmniPMStore {
  // DAG 状态
  loadDAGState(projectId: string): Promise<DAGState | null>;
  saveDAGState(projectId: string, state: DAGState): Promise<void>;
  
  // 质量日志
  appendQualityLog(projectId: string, score: ExpertQualityScore): Promise<void>;
  queryQualityLog(projectId: string, limit?: number): Promise<ExpertQualityScore[]>;
  
  // 项目记忆
  loadProjectMemory(projectId: string): Promise<Record<string, any>>;
  saveProjectMemory(projectId: string, data: Record<string, any>): Promise<void>;
  
  // 审计事件
  appendAuditEvent(projectId: string, event: AuditEvent): Promise<void>;
  queryAuditEvents(projectId: string, filter: AuditFilter): Promise<AuditEvent[]>;
}
```

## 实现策略

| 阶段 | 存储后端 | 场景 |
|------|----------|------|
| 当前 | FileSystemStore | 单用户 CLI |
| Phase 1 | SQLiteStore (better-sqlite3) | 多项目本地 |
| Phase 2 | PostgresStore / Cloudflare D1 | 多用户服务端 |

## 迁移路径

1. 定义 `OmniPMStore` 接口
2. 实现 `FileSystemStore`（封装现有逻辑）
3. 所有 DAG 操作通过 Store 接口，不直接读写文件
4. 切换后端只需替换 Store 实现
